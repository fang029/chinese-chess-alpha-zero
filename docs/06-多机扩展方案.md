# 多机扩展方案(预留,本次未实现)

> 本文档描述把自我对弈扩展到 **多台机器** 的架构与代码扩展点。当前代码是
> **单机多卡**(128 核 + 4×A10,见 docs/03),已能 CPU+GPU 双满载。多机是
> 进一步把自弈吞吐扩到 N 台的方案,适合用户能开 8 台同配置机器时。
> **本次只设计 + 预留扩展点,未写网络通信代码。**

## 为什么多机是最大的吞吐杠杆

AlphaZero 自弈天然可水平扩展:每台机器独立自弈产棋谱,汇总到一处训练。
DeepMind 用数千台。对本项目:

| 方案 | 自弈吞吐 | 说明 |
|---|---|---|
| 单机 128 核 + 4×A10 | 基准(已实现) | CPU 产请求 + GPU 攒批前向 |
| 8 台同配置 + 分布式汇总 | ~8× | 棋谱汇总到 master 统一训练 |

8 台分布式是把达标时间从"1 天"压到"几小时"的唯一线性手段,且不依赖单机堆料。

## 架构:master / slave

```
        ┌─────────── master(1 台)───────────┐
        │  - 持有训练网络,消费汇总的样本训练   │
        │  - 每轮把新权重广播给所有 slave        │
        │  - 落盘 checkpoint / metrics          │
        └───────────────┬───────────────────────┘
                         │ 样本上行 / 权重下行(TCP)
        ┌────────────────┼────────────────┐
     slave1            slave2     ...    slaveN
     (各 128 核 + 4×A10,本地多卡自弈)
     - 每台跑一个 MultiGPUInferenceServer + 本地 worker
     - 产出的样本经网络推给 master
     - 定期从 master 拉最新权重,刷新本地推理服务
```

## 代码扩展点(改动隔离在一层)

当前 `ParallelSelfPlay` 的样本出口是本地 `self.sample_queue`(进程内 mp.Queue)。
多机版只需把这个 queue 换成**网络队列**,其余(worker、推理服务、MCTS)全不变。

具体预留点:

1. **样本上行**:`pipeline_parallel.py` 主循环用 `parallel.collect_games()` 从
   `sample_queue` 取样本。多机版:slave 端把样本 `put` 到一个网络客户端(socket/
   zmq/Redis list),master 端从网络服务端 `get`。
   - 新增 `xiangqi/distributed_selfplay.py`:`NetSampleClient`(slave 推样本)、
     `NetSampleServer`(master 收样本),接口模仿 mp.Queue 的 put/get。
   - `ParallelSelfPlay` 增加 `sample_sink` 参数(默认本地 queue,多机传网络 client)。

2. **权重下行**:master 每轮 `publish_weights` 后,把权重文件推给各 slave;
   slave 收到后调本地 `server.publish_weights`(已有的热重载机制直接复用)。
   - 复用现有"写文件 + 版本号热重载":master 通过网络把权重文件同步到 slave
     的 `weights_path`,slave 的 `MultiGPUInferenceServer` 自动热重载(已实现)。

3. **slave 入口**:新增 `xiangqi/slave.py`,启动本地 `MultiGPUInferenceServer`
   + worker,样本推给 master,轮询拉权重。master 仍用 `pipeline_parallel.py`
   的训练循环,只是样本来源换成 `NetSampleServer`。

## 通信选型建议

- **同内网 8 台**:用 ZeroMQ(PUSH/PULL 收样本,PUB/SUB 广播权重)或裸 TCP socket。
  样本是 numpy 数组,序列化用 `np.save` 到 BytesIO 或 msgpack-numpy。
- **跨网/简单**:用 Redis(slave LPUSH 样本,master BRPOP;权重放对象存储/共享盘)。
- 权重文件 ~30MB(64×6)/ ~130MB(128×10),每轮广播一次,带宽要求不高。

## 实现顺序(后续真要做时)

1. `distributed_selfplay.py`:NetSampleClient/Server(先 TCP socket + numpy 序列化)。
2. `ParallelSelfPlay` 加 `sample_sink` 注入点(默认不变,多机传网络 client)。
3. `slave.py`:slave 进程入口(本地多卡自弈 + 样本上行 + 权重拉取)。
4. master 端:`pipeline_parallel.py` 加 `--master` 模式,样本源换 NetSampleServer。
5. 验证:1 master + 2 slave 同内网跑通,样本正确汇总、权重正确下发。

## 注意

- **样本时效**:slave 用的权重比 master 略旧(异步),AlphaZero 容忍这种 off-policy
  滞后(本就有 replay buffer)。无需严格同步。
- **容错**:slave 掉线不影响 master(样本少了而已);slave 重连后继续推。
- **网络形状一致**:所有机器的 `--channels/--blocks` 必须相同(权重要互通)。
- 多机不是必须——单机 128 核 + 4×A10 已能双满载。多机是"还想更快"时的选项。
